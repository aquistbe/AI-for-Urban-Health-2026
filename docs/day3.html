Learning Objectives

  1. Distinguish between AI chatbots and AI agents and explain why the difference matters
  2. Describe the five components of context engineering and why it is critical for effective AI use
  3. Explain the Model Context Protocol (MCP) and how it enables tool integration

Lecture: Agents, Context, and Tools

Chatbots vs. Agents

The AI landscape is rapidly moving from chatbots (reactive, single-turn, limited to text) to agents (proactive, multi-step, tool-using):

Feature Chatbot Agent
Interaction Single turn Q&A Multi-step task completion
Tools None (text only) Can use tools (search, code, APIs)
Memory Limited to conversation Can maintain context across tasks
Autonomy Responds only when asked Can take initiative, plan, and execute
Error handling Stops at confusion Can retry, adapt, self-correct

Why this matters for public health: Agents can do things like “pull the latest flu surveillance data, compare it to last year, draft a summary report, and email it to the team”—tasks that involve multiple steps, tools, and decision points.

Context Engineering: The 5 Components

Context engineering is the practice of designing the information environment that an AI system operates within. It is more fundamental than prompt engineering. The five key components:

  1. System prompt / Instructions: What the AI should do, its role, its constraints
  2. User input: The specific request or query
  3. Retrieved knowledge: Relevant documents, data, or information pulled in for the task (RAG — Retrieval-Augmented Generation)
  4. Tool definitions: What tools the AI can use and how to use them
  5. Conversation history / Memory: What has happened before in the interaction
TipThink of It Like Briefing a New Employee

Context engineering is like preparing a briefing packet for a highly skilled new employee on their first day. You would not just say “do your job.” You would provide: their role description (system prompt), the specific task (user input), relevant background materials (retrieved knowledge), access to the tools and systems they need (tool definitions), and notes from previous meetings (conversation history).

Model Context Protocol (MCP)

MCP is an emerging open standard (created by Anthropic) that allows AI models to connect with external tools and data sources through a standardized interface. Think of it as a “USB port for AI”—a universal way to plug in capabilities.

Examples of MCP tools:

  • File system access (read/write documents)
  • Database queries
  • Web search
  • Calendar and email integration
  • Code execution
  • API calls to external services

Agent Skills

A Skill is a reusable folder of expertise you hand the agent. At minimum it is a SKILL.md file — a Markdown document with YAML frontmatter (a name and a description) and instructions in the body — plus optional scripts/ (code the agent can run) and references/ (documents loaded only when needed). When a task matches the description, the agent loads the skill on demand and follows it.

If MCP is how an agent reaches tools, a Skill is how it knows your way of doing a task. Where a prompt is a one-off instruction, a Skill is durable and reusable: write your de-identification checklist, your report format, or your STROBE/CONSORT reporting steps once, and the agent applies them every time — across chats, Claude Code, and Cowork.

You can write a SKILL.md by hand, or use the Skill Creator (a “skill that builds skills”) — it’s pre-installed in Claude Desktop and Cowork, and installable in Claude Code (/plugin install skill-creator). On claude.ai, upload a zipped skill folder under Settings → Customize → Skills.

Public-health skill ideas: a “de-identify before sharing” checklist, a plain-language health-communication rewriter, a grant Specific-Aims formatter, or a data-dictionary generator.

Agentic Harnesses: Cowork, Claude Code, Codex, and More

The same core idea — a model plus tools running in a loop — ships in several “harnesses” that differ mainly in where you work and how much autonomy they take. (Recall the three ways to work with AI from Day 1.)

Harness What it is Best for Notes
Claude Code (Anthropic) Terminal/CLI agent Coding, data analysis, file automation in a repo Reads CLAUDE.md; strong on coding benchmarks; Anthropic models
Cowork (Anthropic) Desktop app agent Non-coders: files, connectors, Skills, scheduled tasks Runs each task as its own session; paid plan
Codex / Codex CLI (OpenAI) Terminal agent Long, “walk-away” autonomous runs (“goal mode”) Strong Docker sandboxing; OpenAI models
Google Antigravity Agent-first IDE + CLI/SDK Orchestrating multiple agents; browser-testing what it builds Free public preview; Gemini models (Gemini CLI folded into it)
OpenCoworker Open-source desktop agent Privacy-sensitive work; bring-your-own model Free/self-hostable; use your API key or a local Ollama model

The differences that matter for choosing: surface (terminal vs. desktop vs. IDE), autonomy (one step vs. hours of unattended work), model lock-in (vendor-only vs. bring-your-own), and data control (cloud vs. fully local). For sensitive public-health data, weigh the last two heavily.

Scheduled Tasks

Agentic tools increasingly run on a schedule, not just on demand. In Cowork, you describe a task once, pick a cadence (hourly, daily, weekdays, weekly) or an event trigger (a new file, a webhook), and it runs automatically — each run with access to your files, connectors, and Skills. This turns a one-off (“summarize this week’s 311 complaints”) into standing automation (“every Monday, pull the new data, update the chart, and draft the email”). Schedule only work you’d be comfortable having run with a human reviewing the output — not irreversible actions.

Verification Is the Bottleneck

Agents generate far faster than you can check, so verification — not generation — becomes the limiting step and the real skill. Fluent output is not the same as correct output, and the higher the stakes, the higher your burden of proof. One of the most reliable uses of a model is to turn it on its own work: ask it to be an adversarial reader (“argue why this analysis is wrong”), to find the weakest claim, the missing control, or the unchecked assumption in your code, draft, or study design — then you decide.

Activity 1: Context Engineering Exercise

Activity 1: Context Engineering Exercise

Duration: ~20 minutes | Format: Small group design exercise

Goal: Design a context package for a specialized public health AI assistant.

Scenario

You are designing a “Neighborhood Health Assessment AI Assistant” for the Philadelphia Department of Public Health. This agent should be able to:

  • Pull demographic and health data for any Philadelphia neighborhood
  • Compare indicators across neighborhoods
  • Generate assessment reports with visualizations
  • Identify health equity concerns
  • Recommend evidence-based interventions

Your Task

In your small group, design the five context components for this agent:

1. System Prompt / Instructions

  • What role should the agent assume?
  • What are its core capabilities?
  • What constraints should it have? (e.g., never provide individual-level data, always cite sources, flag uncertainty)

2. User Input Examples

  • Write 3 example queries a health department analyst might ask this agent

3. Retrieved Knowledge

  • What documents, datasets, or knowledge bases should this agent have access to?
  • How often should they be updated?

4. Tool Definitions

  • What tools should this agent be able to use? (e.g., database query, mapping, chart generation, report writing)

5. Conversation History / Memory

  • What should the agent remember across conversations?
  • What should it forget (for privacy/security)?

Share Out

Each group presents their context design in 2 minutes. Class discusses: What did groups prioritize differently? What components were hardest to define?

Activity 2: Claude Code CLI Walkthrough

Activity 2: Claude Code CLI Walkthrough

Duration: ~25 minutes | Format: Instructor demonstration (follow along if you have Claude Code installed)

Goal: See how a command-line AI tool can perform data analysis, write code, and generate documents directly on your computer.

What is Claude Code?

Claude Code is a command-line interface (CLI) tool from Anthropic that lets you interact with Claude directly from your terminal. Unlike the web-based chat, Claude Code can:

  • Read and write files on your computer
  • Execute code and see results
  • Navigate your file system
  • Chain multiple operations together
  • Work with local data that never leaves your machine (though prompts are still sent to Anthropic’s servers)

Demo Outline

The instructor will demonstrate:

1. Data Analysis and Visualization

claude "Load the Philadelphia HVI dataset from data/philadelphia_hvi.csv.
       Summarize it and generate a bar chart of the 10 census
       tracts with the highest hvi_score. Save the chart
       as hvi_top10.png"

2. Document Generation

claude "Read the analysis results in output/ and draft a
       2-page executive summary for city council. Include
       the key findings, a table of the most vulnerable
       tracts, and three policy recommendations.
       Save as executive_summary.docx"

3. Multi-Step Workflow

claude "Download the latest flu surveillance data from CDC,
       compare Philadelphia's rates to the national average,
       create a trend visualization, and draft a weekly
       update email for the department director"

Key Takeaways

  • CLI tools are powerful for integrating AI into existing workflows
  • They work with your local files and tools
  • They enable automation of repetitive analysis-report-communicate cycles
  • They require some comfort with the command line (but AI can help with that too)
NoteInstructor Walkthrough: Running the Claude Code Demo Live

This is a step-by-step guide for the instructor to run the demo end-to-end on a real dataset. It uses a small sample Heat Vulnerability dataset so every command actually executes. Total live time: ~20–25 minutes. (Verified against Claude Code documentation, 2026; the CLI changes often, so test once the day before.)

Before class (15 minutes, once)

  1. Install (needs Node.js v18+; check with node --version):

    npm install -g @anthropic-ai/claude-code
claude --version          # confirm it installed

    Do not use sudo. If you hit permission errors, use nvm or reconfigure npm’s global prefix.

  2. Authenticate. Run claude once; a browser opens to approve login. This works with a Claude Pro/Max or Team/Enterprise subscription (recommended for teaching), or with an API key from the Console.

    Cost gotcha: if ANTHROPIC_API_KEY is set in your shell, it overrides your subscription and you’ll be billed per-token. For a subscription demo, unset ANTHROPIC_API_KEY first.

  3. Build the demo folder and drop in the data (download philadelphia_hvi.csv — the real OpenDataPhilly HVI: 384 Philadelphia tracts, 2023):

    mkdir -p ~/cli-demo/data && cd ~/cli-demo
# move the downloaded file into ~/cli-demo/data/philadelphia_hvi.csv

  4. Dry-run it once so there are no surprises live, then /clear before class.

Live demo (project on screen, narrate as you go)

Step 0 — Start in the demo folder (1 min). Open the terminal in ~/cli-demo and launch the REPL:

cd ~/cli-demo
claude

Point out the prompt is now waiting, and that Claude can see this folder. Type /help to show the slash commands, then /cost to show that usage is tracked.

Step 1 — Explore the data (read-only; ~4 min). Paste:

Look at data/philadelphia_hvi.csv. How many records are there, what is the unit of analysis, what columns are present, and are there any data-quality issues? Don’t write any files yet — just summarize.

What to narrate: Claude reads the file without asking permission (reads are allowed by default), but it has not changed anything. This is the safe way to start: explore before you act. Tie back to context engineering (Activity 1) — it only knows what’s in the folder plus your instructions.

Step 2 — Generate a chart (first permission prompt; ~6 min). Paste:

Create a horizontal bar chart of the 10 tracts with the highest hvi_score, labeled by name10 (the tract number). Use Python (matplotlib), save it as figures/hvi_top10.png, and create the figures/ folder if needed.

What to narrate: Claude now wants to write a script and run it, so it pauses and shows you the command and the file it will create. Approve it in front of the class — this is the human-in-the-loop checkpoint and the single most important thing to show. Open figures/hvi_top10.png afterward.

Step 3 — Draft a brief from the findings (~5 min). Paste:

Using what you found, draft a one-page plain-language brief for City Council on heat vulnerability, with the top 5 tracts (by name10) in a table and three policy recommendations. Save it as brief.md.

What to narrate: Same approval step for the new file. Then open brief.md and read a recommendation aloud, then push back: ask the class “is that number actually in our data?” Open the CSV and check. This is the verification-is-the-bottleneck point from earlier — fluent output still needs checking, especially before anything reaches a decision-maker.

Step 4 — Show the controls (~3 min).

  • /permissions — how to pre-approve or block specific commands.
  • Press Shift+Tab to cycle to plan mode (read-only): Claude proposes a plan but cannot edit. Good for “think before you touch my files.”
  • /clear to wipe context; /cost to show total session cost; /exit (or Ctrl+D) to quit.

Alternative: run it inside Positron (recommended for R/Python users)

Claude Code is still the CLI here — you just host it in Positron (Posit’s R/Python IDE) instead of a bare terminal. For a data-science audience this is usually the better venue: students stay in a familiar environment, and Claude’s edits and outputs appear right in the IDE.

  1. Open the demo folder in Positron: File → Open Folder → ~/cli-demo.
  2. Open the integrated terminal (Terminal → New Terminal, or Ctrl+`) and run claude there. Claude Code auto-detects the IDE (via a lock file in ~/.claude/ide/), so it knows which file you have open and shows its edits as diffs in the editor — which is much clearer to project than a terminal diff.
  3. Keep it in your language. Add to your prompts: “Use R and ggplot2; save the script as a .R file and run it with Rscript” (or “use Python”). Run the same four steps as above.
  4. Verify in the IDE. Open philadelphia_hvi.csv in Positron’s data viewer, let the chart land in the Plots pane, and run one line of R/Python yourself to confirm Claude’s numbers — a clean, visible version of the verification step.

Caveat: Claude runs commands in that terminal’s shell, so the R/Python it uses is whatever is on your PATH, not necessarily Positron’s selected interpreter. Confirm Rscript (or your Python environment) works in that terminal before class.

Positron also ships a built-in Positron Assistant (a native in-IDE chat/agent, bring-your-own Claude API key). That’s a different flavor from the agentic-CLI lesson, so keep Claude Code as the tool you demo — but it’s worth mentioning to students as another option.

If something breaks (fallbacks)

  • Can’t install / locked-down laptop: run the same prompts against claude.ai or Google AI Studio by uploading philadelphia_hvi.csv. You lose the local-file/automation angle but keep the analysis narrative.
  • No internet in the room: pre-record the run, or screenshot each step into the slides as a backup.
  • Want a no-prompt, scripted version: claude -p "summarize data/philadelphia_hvi.csv" runs once and exits — useful to show automation, but do the interactive version first so students see the approval prompts.
  • Don’t use --dangerously-skip-permissions in class except to name it and explain why skipping approvals is risky.

The three points to land

  1. Agentic CLIs act on your real files and tools, not just chat text.
  2. The permission prompt is the safety boundary — you stay in the loop on every write and command.
  3. Generation is cheap; verification is the job. Always check the numbers before the output leaves the room.

Activity 4: Google AI Studio Agent Building

Instructions

Option A: Disease Prediction Agent

Build an agent that helps public health analysts assess disease outbreak risk:

System Instructions:

You are an infectious disease risk assessment assistant for the Philadelphia Department of Public Health. Your role is to help analysts evaluate potential outbreak risks based on surveillance data and environmental factors.

When analyzing a potential outbreak: 1. Ask clarifying questions about the data (number of cases, location, time period, demographics affected) 2. Compare to historical baselines 3. Option B: Environmental Health Risk Assessor

You are an environmental health risk assessment assistant. You help public health analysts evaluate environmental exposures and their potential health impacts in Philadelphia neighborhoods.

For each assessment: 1. Identify the exposure pathway (air, water, soil, food) 2. Assess the exposed population (size, vulnerability factors) 3. Evaluate exposure duration and intensity 4. Review known health effects from the scientific literature 5. Recommend monitoring, mitigation, and communication strategies

Always specify the strength of evidence for health effects. Distinguish between established and emerging science.

Activity 5: Local AI with Ollama

Activity 5: Local AI with Ollama

Duration: ~20 minutes | Format: Demonstration + optional hands-on installation

Goal: Understand why local AI models matter for sensitive data and see how to run them.

Why Local Models?

When you use ChatGPT, Claude, or Gemini, your prompts and data are sent to external servers. For many public health tasks, this is fine. But for work involving:

  • Protected Health Information (PHI)
  • Personally identifiable data
  • Pre-publication research data
  • Sensitive agency communications
  • Data covered by IRB protocols

…you may need AI that runs entirely on your own machine, with no data leaving your computer.

What is Ollama?

Ollama is a free, open-source tool that lets you run AI models locally on your laptop. It supports many models, including:

  • Llama 3.x / Llama 4 (Meta) — good general-purpose models
  • Gemma 4 (Google) — lightweight and capable; runs well on a laptop
  • Qwen / DeepSeek — strong open models (note: developed in China; data stays local when run via Ollama)
  • Mistral — fast and efficient
  • Phi (Microsoft) — optimized for smaller hardware

Installation (for those who can install software)

# macOS (Homebrew)
brew install ollama

# Or download from https://ollama.com/download

# Start Ollama
ollama serve

# Pull and run a model (pick one that fits your laptop)
ollama run qwen3          # strong default; try qwen3:4b on 8 GB laptops
ollama run gemma4         # Google's efficient option

System Requirements:

  • At least 8 GB RAM (16 GB recommended)
  • 5–10 GB free disk space per model
  • macOS, Linux, or Windows

Browser-Based Alternatives (for restricted computers)

If you cannot install software on your work computer:

  • Google AI Studio: Free, browser-based, runs Gemini models with reasonable privacy policies
  • HuggingChat: Free, open-source models, browser-based
  • Note: These still send data to external servers but may have stronger privacy protections than consumer products

Demo

The instructor will demonstrate:

  1. Installing and running Ollama
  2. Chatting with a local model
  3. Comparing response quality to cloud-based models (trade-offs: speed, quality, privacy)
  4. When local models are “good enough” vs. when you need cloud models
TipAdvanced companion: agentic workflows on a local model (notebook)

Want to go further than chatting? This Jupyter notebook builds real agentic workflows on a local Ollama model (a laptop-friendly model like Qwen3 or Gemma 4) — everything runs on your machine, free and private. It works through three patterns on a small set of urban-health abstracts (pedestrian safety, heat, air quality):

  1. One structured-output prompt to screen an abstract against inclusion criteria (Pydantic + JSON mode).
  2. A multi-agent crew (CrewAI) — a screener and an extractor collaborating.
  3. A state machine (LangGraph) — the same workflow as an explicit graph with conditional branching.

Plus a tool-calling agent (a stand-in PubMed search + a confidence-interval calculator) that previews what MCP standardizes, and a “try it yourself” exercise. It ties directly to the Day 3 concepts: the agent loop, context engineering, tools/MCP, guardrails, and verification.

Run it in Positron, VS Code, or Jupyter. Install once with pip install ollama crewai langgraph langchain-ollama pydantic. Best as an optional advanced track or a live instructor demo.

NoteFor IT-Restricted Computers

Many government and health department computers restrict software installation. If you cannot install Ollama on your work computer, the browser-based alternatives above provide a middle ground. For truly sensitive work, talk to your IT department about setting up a local AI environment on a secure server.
